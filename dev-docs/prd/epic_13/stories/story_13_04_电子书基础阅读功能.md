# Story 13-04: 电子书基础阅读浏览

> Epic: 与作者共读
> Priority: High

## 概述
为用户提供一个专门打造的阅读器窗口，能够直接打开和排版之前导入的书籍，并完成基本向前向后的翻页浏览。

## 用户故事

### US-1: 打开并浏览书籍
**作为** 用户，
**我希望** 点击书籍列表项即可进入阅读器模式进行阅读排版，
**以便** 能够轻松且专注地看书并阅读书中的文字内容。

## 功能范围

### 包含
- 利用针对不同格式合适的视图引擎或渲染库，绘制电子书具体内容。
- 分解章节长文本或分页等机制实现翻页操作（上一页/下一页、平滑滚动）。
- 定义不同格式的基础阅读打开条件与降级策略：可阅读但不可选中、可阅读但无目录、可阅读但不可聊天。

### 不包含（后续迭代）
- 侧边栏的附属组件及目录系统。
- 标记书签并永久保存的能力。
- 完整的排版自定义能力（字体、字号、主题、行距等高级设置）。

## 补充规则
- “成功打开阅读器”不要求所有格式都使用完全一致的翻页模型；允许某些格式采用滚动阅读，某些格式采用页级翻页。
- 对解析失败、文件损坏、当前设备不支持渲染的场景，必须给出错误态与返回路径。

## 验收标准
- [ ] AC-1: 在图书库中点击绑定或未绑定的一本书，都能够正常跳转至全新独立的电子书阅读界面。
- [ ] AC-2: 系统能够支持正常滚动阅读或点击按钮分屏/翻页。
- [ ] AC-3: 提供一个清晰易懂的“返回”按钮让用户能够结束当前这本书的阅读，退回到上一级书库中。
- [ ] AC-4: 若图书处于 `processing`、`limited` 或 `failed` 状态，阅读器入口和阅读器内部会给出与状态一致的提示，不会展示与实际能力不符的交互。
- [ ] AC-5: 对于无法渲染或打开失败的图书，用户可以明确地返回书库，且不会导致应用主界面卡死或空白。

## 附录：实现约束

### 1. 阅读器引擎引入方式
- `foliate-js` 以 vendoring 形式纳入项目源码，不作为纯 `node_modules` 依赖使用。
- vendored 源码位于前端项目内部，后续对当前页内容提取、搜索、选区、事件桥接等能力的扩展，应在此基础上继续演进。

### 2. 业务接入层约束
- 页面组件不得到处直接引用 `foliate-js` 内部模块。
- 业务层应统一通过阅读器适配层接入，后续目录、进度、页面上下文、引用选区等能力也都应继续挂在适配层上，而不是散落在各个 Vue 组件内。

### 3. 当前格式基础实现边界
- EPUB / PDF：通过 vendored `foliate-js` 打开与渲染。
- TXT：通过自定义适配器转换为与 `foliate` 可消费的统一 `book` 对象。
- “可打开阅读”不等于“自动具备目录、上下文提取、引用交互”等后续能力，后续 Story 需继续基于格式能力做分层降级。

### 4. PDF 资源约束
- PDF 基础阅读依赖 `pdf.js` 的 `cmaps` 与 `standard_fonts` 资源目录。
- 这些资源属于源码运行依赖，应随 vendored 阅读器源码一起保留在仓库中。
- 构建产物中的同名资源目录属于输出结果，不需要提交。

### 5. 当前导航形态
- 本 Epic 当前沿用主界面的状态切换模式进入阅读器，不额外切成独立路由方案。
- 后续 Story 在阅读器中新增目录、聊天侧栏、引用交互时，应继续建立在这一主视图切换结构上，避免重复引入第二套导航体系。
